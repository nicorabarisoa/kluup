<objective>
Fix SC-3 (real local gap, architectural) using the LOCKED approach (user, 2026-06-10): a **pg_cron server sweep (~1 min)**, NO pagehide beacon this pass. Root cause (diagnosed): the room/ghost cleanup for the tab-close path is entirely client-side and driven by SURVIVING peers; when ALL clients close their tabs there is no JS runtime left to prune, transfer host, or delete the room. The only server reaper, `cleanup_dead_rooms()`, uses a 30-minute threshold and only fires opportunistically on room creation. SC-3's tab-close guarantee is not achievable client-side.

Fix:
1. In `supabase/lifecycle.sql`, lower the `cleanup_dead_rooms()` idle threshold from 30 minutes to ~60 seconds, and add/uncomment a pg_cron block that creates the extension and schedules `cleanup_dead_rooms()` every minute.
2. Relax the SC-3 acceptance criterion from ">15s" to "~1 min (server sweep interval)" in `.planning/ROADMAP.md`, and note it in `.planning/phases/03-playtest-quality-fixes/03-UAT.md`.
3. Because this SQL must be APPLIED to the live Supabase DB (Claude cannot run SQL there), the plan ENDS with a BLOCKING human-action checkpoint to run the SQL and confirm the cron job exists — exactly like plan 03-03 did for its migration.

Purpose: SC-3 — a room with zero connected players is automatically deleted (within ~1 min via the server sweep).
