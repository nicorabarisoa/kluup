<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anon client → Postgres (RLS) | Anonymous game traffic relies on open anon policies on rooms/players/votes; any tightening silently breaks the game |
| authenticated user → user_session_stats | Per-user stats rows must be readable/writable only by their owner |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering | RLS policies on rooms/players/votes | high | mitigate | Migration is additive-only — it never DROPs or CREATEs any policy on rooms/players/votes. Verify scripts in both tasks assert the open anon SELECT policies remain present and that no `ALTER TABLE rooms` / `DROP POLICY ... ON rooms|votes` appears. Full AUTH-04 smoke test runs in plan 02-04. |
| T-02-02 | Information Disclosure | user_session_stats | high | mitigate | RLS `USING (auth.uid() = user_id)` on SELECT/INSERT/UPDATE prevents any user reading another user's stats. No DELETE policy means anon/other users cannot delete rows either. |
| T-02-03 | Tampering | players.host_id column | medium | accept | Migration does not touch host_id. CLAUDE.md notes prod has it NOT NULL; ADD COLUMN IF NOT EXISTS on a different column cannot conflict. Verify script asserts no `ALTER ... host_id`. |
</threat_model>
