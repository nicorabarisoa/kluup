## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, stat badge internal padding (`px-2 py-0.5`) |
| sm | 8px | Compact element spacing, gap between stat badges |
| md | 16px | Default element spacing, section padding top (`pt-8 → 32px`) |
| lg | 24px | Card internal padding (`p-6`), section gaps |
| xl | 32px | Page top padding (`pt-8`), layout vertical gaps |
| 2xl | 48px | Major section breaks between profile blocks |
| 3xl | 64px | Not used — page layout stays compact on mobile |

Exceptions:
- Game screen horizontal padding: 20px (matches existing `padding: '0 20px'` in `GameScreen`)
- Footer padding: `12px 20px 32px` (existing pattern — keep for sign-in CTA footer on end screen)
- Touch targets: minimum 44px height for all tappable elements (buttons, session history rows)
- Profile page section max-width: `max-w-md` (448px) for mobile-first centered column. May use `max-w-4xl` + `md:grid-cols-2` grid for per-session history list only if content warrants it (ASSUMPTION: start with `max-w-md`, evaluate at implementation — mark as assumption A-01)

**Source:** existing `GameScreen`, `EndScreen`, `app/page.tsx` measurements.

## Layout Contracts

### Surface A — End Screen Sign-in CTA (anonymous users only)

**Location:** Inside the existing `EndScreen` component, rendered after the players/personal-stats list and before the `fr.end.thanks` closing line. Shown only when `isSignedIn === false`.

**Structure:**
```
[surface card — rounded-2xl p-4, background C.surface, border: 1px solid C.a + '33']
  [heading — 14px, weight 800, color #FFFFFF, font-display]
  [body copy — 12px, color C.muted, font-body, margin-top 4px]
  [CTA button — PrimaryBtn, accent C.a, margin-top 12px, full width]
```

**Visual:** Matches the existing group-title card pattern (rounded-3xl p-6 text-center) but smaller (rounded-2xl p-4, no text-center forced). The card has a subtle accent border (`#FF3C6F33`) to draw the eye without dominating. Not full-bleed — sits within the `max-w-md` content column.

**Interaction:**
- Tap CTA → `supabase.auth.signInWithOAuth({ provider: 'google' })` with `redirectTo` set to the current end-screen URL so the user returns after OAuth
- No animation on the CTA card itself — the existing `active:scale-95 transition-transform` on PrimaryBtn is sufficient

**Visibility rule:** `isSignedIn` prop derived from the same `getUser()` call already present at the game page root (Phase 4). Pass it down to `EndScreen`. The CTA block is conditionally rendered: `{!isSignedIn && <SignInSaveCTA />}`.

### Surface B — Profile Page (`/profile`)

**Route:** `app/profile/page.tsx` — new server component with `'use client'` only where auth state is needed.

**Layout:** Centered column, `max-w-md mx-auto px-5`, matching the game screen layout. Top bar identical to landing page (logo left, auth chip right, LangSwitch).

**Page structure (top to bottom):**

```
[Top bar — logo + auth chip (sign-out button) + LangSwitch]

[Hero block — padding-top 32px]
  [Avatar circle — 56px diameter, C.surface background, player initial in --font-display]
  [Display name — 20px extrabold, --font-display, margin-top 8px]
  [Subtitle — 12px, C.muted, "X sessions played"]

[Archetype block — rounded-2xl p-5, C.surface bg, margin-top 24px]

  [Archetype name — 28px extrabold, --font-display, C.a color, margin-top 4px]
  [Trait bars — top 3 traits, each row: label (12px) left + bar (flex-1) + % (12px) right]
    [Bar fill — 6px height, rounded-full, C.a for #1 trait, C.b for #2, C.c for #3]
  [Fallback — if tag_scores all zero: "Une simple personne" in C.muted, no bars shown]

[Cumulative stats block — rounded-2xl p-5, C.surface bg, margin-top 16px]
  [4-cell grid (2×2) of stat numbers]
    [Cell: big number (28px extrabold --font-display C.a) + label (12px C.muted --font-body)]
    Cells: Sessions played · Times designated · Confessions · Times volunteered

[Session history section — margin-top 24px]
  [Section header — 16px extrabold --font-body, margin-bottom 12px]
  [Session list — flex-col gap-2]
    [Session row — rounded-2xl p-4, C.surface bg]
      [Row top: group-title name (14px weight-800 --font-display C.a) + date (12px C.muted) — space-between]
      [Row bottom: stat chips (same badge style as EndScreen) — flex-wrap gap-1.5 mt-2]
        Chips: designated N× (C.a) · confession N× (C.b) · volunteer N× (C.c) — only shown if > 0

[Empty state — when 0 sessions]
  [Centered illustration-free block, mt-16]
  [Heading — 20px extrabold --font-body]
  [Body — 14px C.muted --font-body, mt-2]
  [CTA — GhostBtn linking to / (landing) — mt-8]
```

**Auth guard:** If user is not signed in and visits `/profile`, redirect to `/` (landing). Implement via `redirect('/')` in the server component after `supabase.auth.getUser()`.

### Surface C — Profile Entry Point (landing page)

**Location:** Top bar of `app/page.tsx`, within the existing signed-in auth chip button.

**Current state (Phase 4):** The signed-in chip shows `{firstName} · Se déconnecter` and calls `handleSignOut` on click.

**Phase 5 change:** Add a "Mon profil" link next to the existing chip. Do not remove the sign-out button. Placement: `firstName chip` | `Mon profil link` | `LangSwitch`.

**Alternatively** (ASSUMPTION A-02, mark for human review): repurpose the signed-in chip into a two-action element — clicking the name navigates to `/profile`, and a separate smaller `×` or "déco" inline action signs out. Pick whichever pattern avoids adding a third item to the top bar.

**Preferred approach (default):** Add a `Link href="/profile"` styled as a ghost chip (same `px-2.5 py-1.5 rounded-xl` dimensions as auth chip, `C.surface` bg, `C.border` border) with the profile i18n key. The sign-out action stays on the chip. Total top-bar width stays within `max-w-5xl`.

## Interaction Contracts

### Sign-in CTA on End Screen

| State | Behaviour |
|-------|-----------|
| Default | Card visible below personal stats list, above `fr.end.thanks` line |
| Tap CTA | `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: window.location.href } })` — browser navigates to Google |
| After OAuth return | User is now signed in; `isSignedIn` becomes true; CTA card disappears (conditional render) |
| Already signed in | CTA card not rendered — zero height, no placeholder |

### Profile Page States

| State | Behaviour |
|-------|-----------|
| Loading | Full-page loading state: centered spinner or skeleton rows in `C.surface` at correct heights (no layout shift) |
| Loaded, 0 sessions | Empty state block (heading + body + CTA link to `/`) |
| Loaded, N sessions | Archetype block + cumulative stats + session list (newest first) |
| Load error | Error message (`profile.error_load`), `GhostBtn` to retry (re-fetches, no page reload) |
| Not signed in | `redirect('/')` on server — user never sees the profile page unauthenticated |

### Session History Row

| Interaction | Behaviour |
|------------|-----------|
| Tap row | No action — rows are display-only. No expand/collapse in this phase. (ASSUMPTION A-03: keep simple, validate with user) |
| Overflow: long group title | Single-line, truncate with `text-ellipsis overflow-hidden whitespace-nowrap` |
| Overflow: many stat chips | `flex-wrap` — chips wrap to next line (matches end screen pattern) |

### Profile Entry (Landing)

| State | Behaviour |
|-------|-----------|
| Not signed in | No profile link shown |
| Signed in, auth loading | Chip skeleton / hidden (matches existing `authLoading` guard) |
| Signed in | Profile link chip visible next to auth sign-out chip |
| Tap profile link | `router.push('/profile')` |

## Archetype Block — Detailed Spec

The archetype block is the most novel visual element in this phase.

```
[Container — rounded-2xl p-5, background C.surface, border: 1px solid C.border]
  [Top label — "Ton archétype" (fr.archetypes.card_title)]
    font: --font-body, 12px, weight 500, color C.muted, uppercase, letterSpacing 0.06em
  [Archetype name — e.g. "Le Joker"]
    font: --font-display, 28px, weight 800, color C.a (#FF3C6F), margin-top 4px
  [Trait bars — shown only when tag_scores is non-zero]
    [Row: flex items-center gap-2, margin-top 12px for first, gap-y 8px between rows]
      [Trait label — 12px, weight 500, --font-body, color C.muted, min-width 80px]
      [Bar track — flex-1, height 6px, rounded-full, background C.border]
        [Bar fill — height 6px, rounded-full]
          Fill color: trait #1 → C.a (#FF3C6F), trait #2 → C.b (#7B2FFF), trait #3 → C.c (#FFD600)
          Width: percentage as % of max(all 6 traits), min 4px so bar is always visible
      [Percentage — 12px, weight 800, --font-body, color #FFFFFF, min-width 32px, text-right]
  [Fallback state — tag_scores all zero or null]
    Single line: "Une simple personne" (fr.archetypes.archetype_fallback)
    font: --font-body, 14px, color C.muted, margin-top 8px
```

**Bar fill color assignment:** sorted by score descending. Rank 1 gets `C.a`, rank 2 gets `C.b`, rank 3 gets `C.c`. Only top 3 shown; ties broken by trait key order (alphabetical) for determinism.

**Bar width calculation:** Each bar's width = `(traitScore / maxTraitScore) * 100%`, clamped to [4%, 100%]. This makes bars relative to the player's own strongest trait, not absolute points.

## Open Questions

> **RESOLVED 2026-06-12** — user confirmed all three defaults (A-01: `max-w-md`, A-02: name chip navigates to `/profile`, A-03: display-only rows). These are now locked decisions, not assumptions.

Items that could not be derived from the codebase or REQUIREMENTS.md and require human decision before implementation:

### A-01: Profile page layout width
**Question:** Should the session history use `max-w-md` (448px, matching game screens) or `max-w-4xl` with a 2-column grid (`md:grid-cols-2`) for history rows on desktop?
**Options:**
1. `max-w-md` throughout — simpler, consistent with game screens, slightly cramped on desktop but the app is mobile-first
2. `max-w-4xl` for history section with `md:grid-cols-2` — matches the landing page pattern, better use of desktop space
3. `max-w-2xl` (672px) as a middle ground — wider than game screens, narrower than landing
**Default chosen:** Option 1 (`max-w-md`) — overridable.

### A-02: Profile entry point on landing
**Question:** How should the profile link be surfaced in the top bar for signed-in users?
**Options:**
1. Add a separate "Mon profil" ghost chip next to the sign-out chip (top bar gets 3 items: name chip, profile chip, LangSwitch)
2. Make the name chip a link to `/profile` and keep sign-out as the trailing action inside the same chip (e.g. `{firstName} → profil · déco`)
3. Replace the sign-out chip with a profile chip that has an inline sign-out button (`⋮` dropdown or secondary tap)
**Default chosen:** Option 2 — name chip navigates to `/profile`, sign-out stays in the chip as a separate span tap target.

### A-03: Session history row interactivity
**Question:** Should tapping a session history row expand to show more detail?
**Options:**
1. No interaction — display-only rows (simplest, ships fastest)
2. Expand to reveal the theme played and a text summary of the group title description
3. Navigate to a `/profile/session/{id}` detail page
**Default chosen:** Option 1 — display-only rows.
