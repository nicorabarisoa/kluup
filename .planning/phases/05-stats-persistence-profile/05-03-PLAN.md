Wire stats persistence and the anonymous-conversion CTA into the game page's `ended` phase: add auth state at the game-page root, an idempotent upsert that writes the signed-in player's session stats, a single `onAuthStateChange` listener that (a) refreshes Realtime auth on `TOKEN_REFRESHED` and (b) retroactively saves after the CTA sign-in, and the `SignInSaveCTA` card + "✓ saved" line inside `EndScreen`.

The game page currently holds NO auth state — the listener added here is brand new and carries three responsibilities through one subscription.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser client → user_session_stats | Each signed-in client writes ONLY its own row; RLS enforces ownership |
| OAuth redirect (`?next=`) → app routing | Untrusted redirect target could be abused for open-redirect / PKCE collision |
| game_state (client) → stats row values | Client-computed counters are written to the DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-07 | Tampering | user_session_stats INSERT (write for another user) | mitigate | Write gates on `user?.id` from server-validated `getUser()`; `user_id` set to `user.id`; RLS `stats_insert_own` (`auth.uid() = user_id`) blocks any cross-user write at the DB |
| T-05-08 | Spoofing | OAuth `redirectTo` open-redirect / PKCE `?code` collision | mitigate | `redirectTo` is ALWAYS `${origin}/auth/callback?next=<same-origin path>` (never a raw page URL); `/auth/callback` already validates `next` is same-origin (RESEARCH Security Domain) |
| T-05-09 | Tampering | Inflated self-stats via client game_state manipulation | accept | Low risk for MVP — no rewards tied to stats; RLS still enforces own-row-only; documented in RESEARCH Security Domain |
| T-05-10 | Denial of Service | Anonymous client attempts the write (RLS reject) | mitigate | Write gated on `user?.id` non-null (Pitfall 2) — anon clients never attempt the insert; failure is silent `console.error`, never blocks the party |
| T-05-11 | Information Disclosure | Retroactive `players.user_id` update leaks across rooms | mitigate | Update scoped `.eq('id', pid)` where `pid = getPlayerId(code)` (this room's own player id); failure swallowed silently (PGRST116 on dead room) |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs; all APIs from the existing `@supabase/supabase-js` dependency (RESEARCH audit) |
</threat_model>
