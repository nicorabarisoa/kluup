<objective>
Close the major UAT gap: an anonymous player who taps the end-screen "Se connecter" CTA and takes several minutes on the Google sign-in screen lands back on `/` (home) with their session stats lost forever. Root cause (diagnosed in `.planning/debug/oauth-return-lands-on-home.md`): `signInWithOAuth` is a full-page redirect that destroys all React state; in a solo room the presence heartbeat stops and the pg_cron sweep deletes the room (cascading `game_state.session_uuid` + `votes`) during the multi-minute sign-in; on return the game page's `init()` finds no room and the silent guard at `app/room/[code]/game/page.tsx:1642` (`if (!roomData) { router.push('/'); return }`) fires before `EndScreen` and its retroactive-save effect can ever mount.

This plan implements the **most robust** of the three diagnosed fix directions (fix direction 1): stash the entire save payload in `localStorage` BEFORE the redirect, and flush it on `SIGNED_IN` from a globally-mounted client component — making the save completely independent of room lifetime and of which page the OAuth round-trip lands on. It also handles fix direction 3 (no silent bounce) explicitly: `init()`'s room-not-found branch is modified to detect a pending-stats stash and defer/suppress the bare `router.push('/')` until the flush completes and the confirmation shows. The SQL TTL exemption (fix direction 2) is handled separately in plan 05-06 as defense-in-depth.

Purpose: PROF-02's promise ("sign in to save YOUR stats") must be honored at the hottest conversion moment even when the room dies during a slow OAuth — currently it is silently broken.
Output: `lib/utils.ts` (stash helpers + 24h staleness TTL), `app/PendingStatsFlusher.tsx` (new global flusher), `app/layout.tsx` (mount it), `lib/i18n.ts` (confirmation string), `app/room/[code]/game/page.tsx` (CTA stashes payload + init() not-found branch no longer silently bounces).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage → user_session_stats upsert | The stashed payload is attacker-controllable client data; it is written under the signed-in user's own RLS scope |
| anonymous client → Supabase Auth (OAuth) | Untrusted full-page redirect round-trip |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-15 | Tampering | localStorage stash forged with arbitrary counts/session_id | accept | Stats are vanity/non-authoritative (CLAUDE.md: "nudge, pas coffre-fort"); the upsert is RLS-scoped to `auth.uid() = user_id` so a user can only write their OWN rows; no privilege or money attached. Same trust model as the existing client-computed write effect. |
| T-05-16 | Spoofing | flush writes user_id from session.user.id (server-validated) not from stash | mitigate | user_id is taken from `session.user.id` (Auth-validated) / `getUser()`, never from the stash; the stash carries no user_id. RLS INSERT policy `auth.uid() = user_id` blocks cross-user writes. |
| T-05-17 | Info disclosure | stash persisting in localStorage after flush | mitigate | `clearPendingStats()` on successful flush; a 24h TTL on read self-expires an abandoned stash; only one short-lived payload, no secrets/PII (pseudo not stored, only counts + theme + group_title key). |
| T-05-18 | DoS | repeated SIGNED_IN events causing duplicate writes | mitigate | `flushingRef` guard + `ignoreDuplicates` on `UNIQUE(user_id, session_id)` + stash cleared on success → at most one effective write per session. |
| T-05-22 | Tampering | a stale stash from a cancelled OAuth flushing onto an unrelated later sign-in | mitigate | `getPendingStats()` returns null and clears the stash when `stashed_at` is older than 24h, so an abandoned stash cannot resurface against a future session. |
</threat_model>
