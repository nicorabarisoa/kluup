<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Google OAuth (via Supabase) | `signInWithOAuth` initiates external IdP flow; PKCE handled by existing callback route |
| client → Supabase Auth server | `getUser()` validates session JWT; `signOut()` clears server session cookie |
| client → players table (Postgres, RLS open) | createRoom host insert writes a player row; `user_id` is self-reported |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-10 | Tampering | OAuth CSRF on sign-in | mitigate | PKCE flow already implemented in `app/auth/callback/route.ts` (Phase 2); this plan only calls `signInWithOAuth({ provider: 'google' })`. ASVS V2. |
| T-04-11 | Spoofing | forged `user_id` in host insert | accept | Documented Phase 4 posture (04-RESEARCH.md Security Domain): RLS open, `user_id` self-reported, no gating depends on it. Phase 5 adds `CHECK auth.uid() = user_id`. Tag: documented-accept, not high. |
| T-04-12 | Tampering | user_metadata (Google name) injection | accept | Used for pseudo pre-fill only, fully editable; DB stores only `pseudo` (maxLength 20, trimmed) and `user_id` (UUID). Display-only. ASVS V5 unchanged. |
| T-04-13 | Session | stale auth state via getSession() | mitigate | Plan mandates `getUser()` (Auth-server validated), forbids `getSession()`. ASVS V3. |
| T-04-14 | Denial of Service | getUser() per render | mitigate | Single mount-time call (empty-dep useEffect); never in render (Pitfall 3). |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (04-RESEARCH.md Package Legitimacy Audit: no new packages in Phase 4). |

No high-severity unmitigated threats. T-04-11 is the only accepted spoofing risk — an explicit, documented Phase 4 design decision with no privilege impact.
</threat_model>
