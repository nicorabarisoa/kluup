## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|

## Component Specifications

### 1. Sign-in Button (signed-out state) — Landing + Join top bars

**Container:** Inserted between the logo/back-button and `<LangSwitch />` in the existing top bar flex row. Flex-shrink-0 to prevent compression.

**Visual spec:**
```
background: #1A1A1A
border: 1px solid #252525
border-radius: 12px (rounded-xl)
padding: 6px 10px (py-1.5 px-2.5) — matches LangSwitch pill
font: 12px / weight 700 / var(--font-body)
color: #FFFFFF
```

**Label (i18n key: `auth.sign_in`):**
- FR: "Se connecter"
- EN: "Sign in"
- ES: "Iniciar sesión"
- DE: "Anmelden"

**Interaction:** `onClick` → `supabase.auth.signInWithOAuth({ provider: 'google' })`. No loading spinner — button becomes disabled + opacity 50% during the OAuth redirect (the redirect happens in < 200ms, spinner not useful).

**Tap target:** Full pill clickable. Min-height 32px (h-8) — consistent with Quit button height.

---

### 2. Signed-in Chip (signed-in state) — Landing + Join top bars

Replaces the sign-in button in the same flex slot. Same outer dimensions as the sign-in button pill.

**Visual spec:**
```
background: #1A1A1A
border: 1px solid #252525
border-radius: 12px (rounded-xl)
padding: 6px 10px (py-1.5 px-2.5)
font: 12px / var(--font-body)
max-width: 140px (overflow ellipsis on long names)
```

**Internal layout:** `[Prénom]·[Se déconnecter]` rendered as a single `<button>` or split into a non-interactive name `<span>` + interactive `<button>`.

Preferred pattern — single `<button>` (simpler, entire chip is the sign-out action):
```
"[Prénom] · Se déconnecter"
         ↑            ↑
     #FFF / 700    #888 / 400
```

The " · " separator uses `color: #555555` as a visual divider, not a functional element.

**Truncation:** Prénom capped at 12 characters before truncation with ellipsis. If `user.user_metadata.full_name?.split(' ')[0]` is longer than 12 chars, truncate to 11 + "…". This keeps the chip from overflowing the top bar on mobile.

**Interaction:** `onClick` → `supabase.auth.signOut()` then `router.push('/')`. No confirmation dialog — sign-out is non-destructive (no data loss; user can sign back in).

### 3. Green Dot Indicator — Quit Button in RoundHeader

Indicates signed-in status inside the game and lobby header. Ultra-minimal — no text, no label, no tooltip.

**Current Quit button spec (existing, unchanged):**
```
padding: px-3 h-8 (12px horizontal / 32px height)
border-radius: rounded-xl (12px)
background: #1A1A1A
border: 1px solid #252525
color: #888888 (muted)
font: 12px / weight 500 / var(--font-body)
```

**Green dot addition:**
```
Position: absolute, top-right corner of the Quit button
Size: 6px × 6px
Shape: circle (border-radius: 50%)
Color: #22c55e
Offset: top: -2px, right: -2px (overlaps the button corner slightly)
Implementation: relative wrapper on <button> + <span aria-hidden="true" style="position:absolute; top:-2px; right:-2px; width:6px; height:6px; border-radius:50%; background:#22c55e" />
```

The Quit button element gains `position: relative` only when the dot is rendered. The dot is conditionally rendered — absent when user is signed out.

**No label, no tooltip, no animation.** The dot is a passive indicator only.

### 4. Pseudo Input Pre-fill — Landing + Join pages

When a signed-in user lands on the landing or join page, the pseudo input is pre-filled with their Google first name.

**Pre-fill logic (from CONTEXT.md D-08 + D-09):**
- Source: `user.user_metadata.full_name?.split(' ')[0]` → first space-separated token
- Fallback: `user.user_metadata.name?.split(' ')[0]`
- Fallback 2: `user.email?.split('@')[0]`
- The field is editable — user may change it freely before submitting

**Visual spec:**
- Input field: no visual change from current design
- Pre-fill hint text (shown when field value matches the pre-filled value and user hasn't edited):
  - i18n key: `auth.pseudo_prefilled_hint`
  - FR: "Nom Google pré-rempli — modifiable"
  - EN: "Pre-filled from Google — you can change it"
  - ES: "Nombre de Google completado — puedes cambiarlo"
  - DE: "Google-Name vorausgefüllt — änderbar"
  - Style: 12px, color `#888888`, `var(--font-body)` — same as existing `storedPseudo` hint

**Priority order for pre-fill (when multiple sources exist):**
1. If `storedPseudo` (kluup_pseudo_ key) exists → use that (returning player's previous pseudo takes precedence)
2. Else if signed in → use Google first name
3. Else → empty field

This preserves the existing reconnect behavior while adding auth pre-fill as a fallback.

## States & Transitions

| State | Landing top bar | Join top bar | Quit button |
|-------|----------------|--------------|-------------|
| Signed out | "Se connecter" pill | "Se connecter" pill | No dot |
| Loading auth state | Nothing rendered (avoid flash) | Nothing rendered | No dot |
| Signed in | "[Prénom] · Se déconnecter" pill | "[Prénom] · Se déconnecter" pill | Green dot |

**Loading state:** Auth state is fetched via `supabase.auth.getUser()` in a `useEffect` on page mount. During this async call, the auth slot in the top bar renders nothing (no placeholder, no spinner) to prevent layout shift. The slot should be `min-width: 0` so the `justify-between` layout stays stable when the slot is empty.

**No flash of sign-in state:** The auth element is conditionally rendered only after `getUser()` resolves. If the user is signed in, the signed-in chip appears; if not, the sign-in button appears. There is never a brief flash of the sign-in button to a signed-in user.

## Copywriting Contract

All strings go through i18n (`useT()` hook). Zero hardcoded text. New i18n keys are prefixed `auth.*`.

| Element | i18n key | FR | EN |
|---------|----------|----|----|
| Sign-in button label | `auth.sign_in` | Se connecter | Sign in |
| Sign-out label in chip | `auth.sign_out` | Se déconnecter | Sign out |
| Pseudo pre-fill hint | `auth.pseudo_prefilled_hint` | Nom Google pré-rempli — modifiable | Pre-filled from Google — you can change it |

**No empty state** for this phase (no list that can be empty).

**No destructive confirmation** for sign-out — it is non-destructive (no data deletion, session is server-side, user re-authenticates immediately).

**Error state (sign-in failure):** OAuth failures redirect silently to `/` (D-04 from Phase 2, existing callback). No inline error message needed in Phase 4 — the existing callback silently absorbs errors.

**Error state (IDEN-02 lookup failure):** If the `user_id` lookup on join fails with a DB error, fall through to the normal insert path. No user-visible error message — degradation is silent and safe (creates a new row instead of reconnecting).

## Interaction Flows

### Flow A — Sign-in from Landing Page
1. User taps "Se connecter" pill in top bar
2. `supabase.auth.signInWithOAuth({ provider: 'google' })` fires
3. Browser navigates away to Google OAuth consent screen
4. After consent, Google redirects to `app/auth/callback/route.ts`
5. Callback sets session cookie, redirects to `/`
6. Landing page mounts, `useEffect` calls `getUser()`, returns signed-in user
7. Top bar renders signed-in chip; pseudo input pre-filled with Google first name

### Flow B — Sign-out from Landing Page
1. User taps "[Prénom] · Se déconnecter" chip
2. `supabase.auth.signOut()` fires
3. `router.push('/')` (or `router.refresh()`) clears React state
4. Top bar renders sign-in button; pseudo input cleared

### Flow C — IDEN-02 Reconnect on Join Page (signed-in, new device)
1. User is signed in, arrives on `/join` with no localStorage entry for the room code
2. `getUser()` returns signed-in user
3. After room lookup, before player insert: query `players WHERE room_id = $id AND user_id = $uid LIMIT 1`
4. If row found: `setPlayerId(code, row.id)` → navigate to lobby/game directly. No confirmation UI.
5. If row not found: fall through to normal insert with `user_id` populated

### Flow D — Joining as Signed-in User (first device or new room)
1. Normal insert path, payload includes `user_id: user.id`
2. Identical to anonymous join but with user_id populated
3. Game flow unchanged

## Implementation Notes for Executor

1. **No new component files required.** All changes are inline modifications to `app/page.tsx`, `app/join/page.tsx`, `app/room/[code]/game/page.tsx` (RoundHeader), and `app/room/[code]/lobby/page.tsx`.

2. **Auth state hook pattern:** Use `useState<User | null>(null)` + `useEffect(() => { supabase.auth.getUser().then(({ data }) => setUser(data.user)) }, [])`. No global auth context — only landing and join pages need auth state in Phase 4. The green dot in RoundHeader gets its own identical hook or receives `isSignedIn: boolean` as a prop.

3. **Green dot prop threading:** The cleanest approach is to check auth state once in the game page root, store as `const [isSignedIn, setIsSignedIn] = useState(false)`, and pass it into `RoundHeader` as a prop. This avoids a second `getUser()` call inside RoundHeader itself.

4. **`supabase.auth.getUser()` is a network call** — do not call it in a render function or in a tight loop. One call per page mount, cached in state.

5. **Top bar right-side refactor:** Both landing and join pages currently end their top bar with bare `<LangSwitch />`. Wrap in `<div className="flex items-center gap-2">{authSlot}<LangSwitch /></div>` where `authSlot` is the sign-in button or signed-in chip (or `null` during loading).

6. **Existing `storedPseudo` logic on join page must not regress.** The auth pre-fill is a new fallback that runs only when `storedPseudo` is null/empty. The priority order (stored pseudo > Google name > empty) must be respected.
