<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Google OAuth (via Supabase) | `signInWithOAuth` initiates external IdP flow; PKCE handled by existing callback route |
| client → Supabase Auth server | `getUser()` validates session JWT; `signOut()` clears server session cookie |
| client → players table (Postgres, RLS open) | join insert and IDEN-02 lookup write/read player rows; `user_id` is self-reported |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | OAuth CSRF on sign-in | mitigate | PKCE flow (code verifier) is already implemented in `app/auth/callback/route.ts` (Phase 2); this plan only calls `signInWithOAuth({ provider: 'google' })`, not a hand-rolled redirect. ASVS V2. |
| T-04-06 | Spoofing | forged `user_id` in player insert | accept | Documented Phase 4 posture (CONTEXT.md Claude's Discretion; 04-RESEARCH.md Security Domain): RLS stays open, `user_id` is self-reported. A forged value only lets an attacker reconnect to an existing row in the SAME room — not a privilege escalation, no gating depends on it. Phase 5 adds `CHECK auth.uid() = user_id`. Tag: documented-accept, not high. |
| T-04-07 | Information Disclosure | un-scoped IDEN-02 lookup leaking another room's player row | mitigate | Lookup is scoped with BOTH `.eq('room_id', room.id)` AND `.eq('user_id', user.id)` (Pitfall 5). Acceptance criterion asserts the exact chain. |
| T-04-08 | Tampering | user_metadata (Google name) injection | accept | Google display name is used for pseudo PRE-FILL only and is fully user-editable before insert; the DB stores only `pseudo` (already user-editable, maxLength 20, trimmed) and `user_id` (a UUID, not injectable). Display-only, ASVS V5 unchanged. |
| T-04-09 | Session | stale auth state via getSession() | mitigate | Plan mandates `getUser()` (Auth-server validated), forbids `getSession()`. ASVS V3. |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (04-RESEARCH.md Package Legitimacy Audit: no new packages in Phase 4). |

No high-severity unmitigated threats. T-04-06 is the only accepted spoofing risk and is an explicit, documented Phase 4 design decision with no privilege impact.
</threat_model>
